# JSON Schema Contract

## Purpose

The JSON output is the enterprise and research integration layer. Text output is for humans. JSON output is for automation, CI/CD, benchmark processing, vulnerability management enrichment, and future dashboards.

## Versioning

Every JSON report must include both the output schema version and the tool binary version:

```json
{
  "schema_version": "0.1.0",
  "tool": "x64lens",
  "tool_version": "0.1.0-dev"
}
```

Schema version and tool version are separate. Tool behavior can change without breaking the schema. Schema changes must be tracked in `CHANGELOG.md`.

## Sprint 5 implemented report shape

Patch 017 implements initial JSON output for:

```bash
x64lens gadgets --format json [--max-depth N] <file>
```

The report is generated from internal analysis records after scanning, exact pattern matching, semantic classification, and scoring. It is not generated by scraping text output.

```json
{
  "schema_version": "0.1.0",
  "tool": "x64lens",
  "tool_version": "0.1.0-dev",
  "target": {
    "path": "./tests/bin/gadgets",
    "format": "ELF64",
    "arch": "x86_64",
    "file_size": 12345,
    "entry": "0x0000000000401000"
  },
  "mitigations": {
    "nx_stack": null,
    "pie": false,
    "relro": "none",
    "rwx_load_segment": false,
    "dynamic_linking": false
  },
  "counts": {
    "raw_candidate_count": 11,
    "ret_count": 10,

    "exact_pattern_count": 11,
    "semantic_candidate_count": 11,
    "unknown_candidate_count": 0,

  "primitive_coverage": {
    "arg_control": true,
    "syscall_num_control": true,
    "syscall_trigger": true,

    "alignment": true,
    "registers": ["rax", "rcx", "rdx", "rsi", "rdi", "rsp", "r8", "r9"]

      "va": "0x0000000000401001",
```

## JSON design rules

- Hex addresses and file offsets are strings.
- Numeric sizes, counts, stack deltas, and scores are JSON numbers when known.
- Unknown stack deltas are represented as `null` with `stack_delta_known: false`.
- Unknown or unscored gadget scores are represented as `null`.
- Booleans are true booleans, not strings.
- Unknown mitigation facts should use `null` or explicit strings such as `"unknown"`, not empty strings.
- Every report must include `limitations`.
- Future SARIF output must be generated from internal analysis records, not from text output.

## Count separation rule

JSON reports must avoid a single ambiguous `gadget_count` field. Use explicit fields:

```text
raw_candidate_count
ret_count
ret_imm16_count
exact_pattern_count
semantic_candidate_count
unknown_candidate_count
scored_candidate_count
```

This preserves the difference between noisy raw byte windows, exact suffix matches, semantic primitive claims, unknown candidates, and scored candidates.

## Current limitations

Patch 017 JSON is intentionally initial and narrow:

- JSON is implemented for `gadgets --format json`, not the future integrated `analyze` command.
- The scanner is exact-suffix based and not a full instruction decoder.
- `nx_stack` may be `null` when `PT_GNU_STACK` is absent.
- Canary and full RELRO are not implemented yet.
- Score values are heuristic and should not be treated as exploitability verdicts.

## Patch 018 validation helper

Patch 018 adds `tools/validate-json-report.py` as a standard-library smoke validator for JSON reports. It does not replace the JSON schema file, but it gives the project a dependency-light way to validate report invariants in WSL, Docker, and CI-style environments.

Recommended controlled fixture check:

```bash
./build/x64lens gadgets --format json --max-depth 4 ./tests/bin/gadgets > /tmp/x64lens-gadgets.json
python3 tools/validate-json-report.py --mode fixture /tmp/x64lens-gadgets.json
```

Recommended generic or system-binary check:

```bash
./build/x64lens gadgets --format json --max-depth 4 /bin/ls > /tmp/x64lens-ls.json
python3 tools/validate-json-report.py --mode system /tmp/x64lens-ls.json
```

The validator checks required fields, count relationships, primitive coverage shape, gadget record fields, score ranges, unknown stack-delta representation, and non-empty limitations. Fixture mode additionally checks the controlled `tests/bin/gadgets` semantic and scoring facts.
